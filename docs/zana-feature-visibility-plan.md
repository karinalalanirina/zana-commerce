# Zana Feature Visibility Plan

| Feature | Keep visible / Hide / Defer | Why | How to control (plan/UI/code) | Notes |
|---|---|---|---|---|
| Official WABA inbox, contacts, templates, storefront, orders | Keep visible | Core launch scope | Plans + normal navigation | Pilot-safe. |
| Twilio official path | Defer | Useful later, not needed in first local/demo setup | Provider settings + onboarding process | Keep off until needed. |
| Baileys / unofficial modes | Hide | Not allowed for pilots/paying clients | Provider allow-list + plan/UI restrictions | Do not delete code. |
| Hotel/restro surfaces | Hide | Out of launch scope | Plans/menu visibility | Keep commerce-only messaging. |
| CTWA ads / Meta ads extras | Defer | Not needed for first 3 to 5 clients | Menu/plan restriction | Roadmap later. |
| Broad bulk-blast behavior | Hide / tightly constrain | Compliance and quality risk | Security settings + operator process | No aggressive send posture at launch. |
| Embedded signup | Defer | Managed SaaS first | Onboarding process | Manual onboarding first. |
| Advanced integrations | Defer | Not core to first revenue path | Plans/menu visibility | Sell later if demand appears. |
| Advanced dashboards | Defer | Zana needs simpler client reporting first | Menu visibility | Thin reporting later. |
| AI autopilot behaviors | Hide | Too risky for commerce launch | Workflow/process | Keep AI human-supervised. |
